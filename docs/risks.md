# 风险与对策

本文档识别 Agent Orchestra 项目的主要风险，并提出相应的对策。

## 风险 1：平台 ToS 合规边界

### 风险描述

各 coding agent 平台（Claude Code、Codex CLI、OpenCode 等）的服务条款可能限制：
- 自动化调用
- 批量使用
- 商业用途
- 账号共享

违反 ToS 可能导致账号封禁、服务中断等严重后果。

### 影响程度

**高**：直接影响用户账号安全和项目可用性

### 对策

1. **明确合规声明**
   - 在 README 中明确声明：仅限本地个人/团队使用
   - 明确不做账号池、不做订阅转售、不做 API 服务

2. **用户教育**
   - 在文档中提醒用户遵守各平台 ToS
   - 提供合规使用指南

3. **技术限制**
   - 不实现账号池功能
   - 不实现订阅转售功能
   - 不提供对外 API 服务

4. **持续监控**
   - 关注各平台 ToS 更新
   - 及时调整项目功能

---

## 风险 2：CLI 接口不稳定

### 风险描述

各平台的 CLI 工具是独立开发的，可能存在：
- 输出格式随版本变化
- 登录态检测方式变化
- 命令行参数调整
- 行为不一致

这会导致 adapter 层需要频繁维护。

### 影响程度

**中**：影响系统稳定性，但可通过适配层缓解

### 对策

1. **适配层设计**
   - 每个平台一个独立 adapter
   - 统一接口抽象（`start / send / stream / stop`）
   - 平台特定逻辑封装在 adapter 内部

2. **版本兼容**
   - 记录各平台 CLI 版本
   - 实现版本检测逻辑
   - 提供版本兼容矩阵

3. **输出解析容错**
   - 实现健壮的输出解析器
   - 处理格式异常情况
   - 提供降级方案

4. **测试覆盖**
   - 各平台 CLI 版本的集成测试
   - 输出格式变化的回归测试
   - 自动化兼容性测试

5. **社区反馈**
   - 建立问题反馈机制
   - 及时响应平台变化
   - 快速发布修复版本

## 风险 3：多 agent 上下文割裂

### 风险描述

每个 agent 有独立的上下文窗口，导致：
- 信息不对称
- 重复工作
- 决策冲突
- 接口不一致

仅靠消息传递可能导致信息失真或丢失。

### 影响程度

**中**：影响协作效率和代码质量

### 对策

1. **Blackboard 机制**
   - 共享 markdown 黑板
   - 任务板（TASKS.md）：记录任务分配和状态
   - 决策日志（DECISIONS.md）：记录重要决策
   - 接口契约（CONTRACTS.md）：记录接口定义

2. **消息设计**
   - 结构化消息格式
   - 消息类型分类
   - 消息历史追踪

3. **上下文同步**
   - Agent 定期读取 Blackboard
   - Orchestrator 广播重要变更
   - 冲突检测与解决机制

4. **最佳实践**
   - 提供协作指南
   - 代码审查流程
   - 接口契约先行

---

## 风险 4：订阅用量限额

### 风险描述

各平台订阅计划通常有用量限制：
- 每日/每月请求数限制
- Token 消耗限制
- 并发请求数限制
- 速率限制

超出限制会导致服务降级或中断。

### 影响程度

**中**：影响系统可用性和任务完成时间

### 对策

1. **配额感知**
   - 监控各平台用量
   - 记录用量窗口
   - 预测用量趋势

2. **智能调度**
   - 优先使用低用量平台
   - 跨平台负载均衡
   - 任务排队机制

3. **降级策略**
   - 用量接近限制时降级
   - 切换到其他平台
   - 暂停非关键任务

4. **用户配置**
   - 允许用户设置用量阈值
   - 提供用量告警
   - 支持手动干预

5. **优化建议**
   - 提供用量优化指南
   - 建议合适的订阅计划
   - 批量处理减少请求

## 应急响应

### 账号封禁

1. 立即停止相关平台的使用
2. 分析封禁原因
3. 联系平台客服申诉
4. 调整使用方式

### 接口变更

1. 检测到接口变更后暂停相关 adapter
2. 分析变更内容
3. 更新 adapter 实现
4. 发布修复版本

### 系统故障

1. 记录故障日志
2. 定位故障原因
3. 实施修复方案
4. 发布事故报告
